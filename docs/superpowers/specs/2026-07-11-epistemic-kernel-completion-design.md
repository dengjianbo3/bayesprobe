# BayesProbe Epistemic Kernel Completion Design

Date: 2026-07-11
Status: Approved; implementation plan ready

## 1. Decision Summary

BayesProbe is a complete epistemic agent paradigm with an explicit applicability
boundary. It is not a universal wrapper for every language task, and it does
not force every question into a finite set of answer hypotheses.

The kernel will be completed by adding:

- Task Admission before Task Framing;
- a strict separation between hypotheses and task-facing answers;
- orthogonal hypothesis competition and coverage semantics;
- open-world exact-answer support with unresolved alternative mass;
- frame-adequacy assessment and bounded hypothesis-space expansion;
- cross-cycle Signal Provenance and Evidence Memory;
- a real ProbeDesigner, Capability Registry, and shared CandidatePoolManager;
- history-aware semantic Hypothesis Evolution;
- task-aware selection, synthesis, and abstention projections;
- versioned migration, public-runner, SDK, config, WebUI, and audit support.

The current phase does not build a private benchmark, a coding Intervention
layer, real search/retrieval adapters, or new public benchmark adapters. Future
validation targets are SWE-bench Verified, Terminal-Bench, and RE-Bench. The
existing synthetic-tested HLE text-MCQ harness remains, but HLE is retained only
as a secondary closed-ended academic capability evaluation and no formal HLE
run belongs to this phase.

## 2. Problem Statement

The current kernel correctly frames model-backed open questions before Belief
State creation, supports exclusive mass and independent credence, and fails
closed instead of creating generic support/refute hypotheses. It still embeds
five assumptions that are too strong:

1. every accepted task can be represented as a finite hypothesis list;
2. every hypothesis is a candidate final answer;
3. an exclusive list generated by a model is exhaustive;
4. repeated model reasoning can be treated as fresh cycle-local information;
5. selecting the highest-ranked hypothesis is a valid projection for every
   task.

The exact-answer failure illustrates the result. If an unseeded question is
framed as candidates `1`, `2`, and `3` while the answer lies outside that list,
normalizing only the named candidates creates false certainty. The correct
behavior is to preserve unresolved alternative mass, detect frame inadequacy,
expand the candidate space, seek fresh discriminating signals, and either
select a supported new candidate or abstain.

A broader question such as "What is the meaning of life?" exposes a different
category error. It does not become epistemically well-defined merely because a
model can generate several philosophical positions. BayesProbe must ask for a
reframed objective and evaluation basis or declare the task outside its scope.

## 3. Paradigm Scope

BayesProbe admits tasks whose answer quality can be revised by evidence,
argument about truth-apt claims, or explicit decision criteria.

High-fit tasks include:

- multiple-choice and exact-answer inference;
- claim verification;
- causal explanation and diagnosis;
- evidence-informed design;
- evidence-informed decisions with explicit criteria.

Tasks that normally require reframing include:

- unrestricted normative or existential questions;
- decisions without criteria or stakeholder preferences;
- prompts whose requested output is too ambiguous to validate.

Tasks outside the current paradigm include:

- pure creative generation;
- translation or formatting with no material uncertainty;
- mechanical transformations that do not benefit from belief revision.

Being complete means Task Admission, framing, belief revision, evolution,
projection, and abstention form one coherent method. It does not mean the
method pretends to be useful for every language task.

## 4. Core Invariants

The completed kernel preserves the existing controller/core, signal/evidence,
and belief/answer separations and adds these invariants:

1. No Belief State exists before an admitted TaskFrame is validated.
2. A hypothesis is a revisable claim relevant to the answer; it is not always
   an answer candidate.
3. Competition among named hypotheses and coverage of the possible hypothesis
   space are independent properties.
4. Model-generated candidate count never proves exhaustiveness.
5. Named exclusive mass never silently consumes unresolved alternative mass.
6. New hypothesis proposals are not Evidence Events.
7. Discovery evidence cannot immediately confirm the hypothesis it caused the
   system to create.
8. Exact repeats and correlated restatements do not accumulate as independent
   evidence across cycles.
9. A probe declares its epistemic purpose and required capability.
10. An unavailable capability is visible and cannot be impersonated by model
    reasoning.
11. Core-produced probe candidates are mandatory input to the next cycle.
12. Projection cannot create evidence or mutate belief.
13. Failure to satisfy an Answer Contract yields abstention or
    `answer_unavailable`, never a generic top-hypothesis fallback.

## 5. Target Runtime

```text
Question + Task Context
  -> Task Admission
      -> admitted
      -> needs_reframing
      -> out_of_scope
  -> Task Framing
  -> TaskFrame + Initial HypothesisFrame
  -> BeliefState + FrameState + EvidenceMemorySnapshot
  -> Seed Signal Inbox with any pre-cycle Passive External Signals
  -> Probe Design
  -> Candidate Pool Management
  -> Probe Selection
  -> Capability-Aware Probe Execution
  -> Join Active results and in-cycle Passive External Signals in Signal Inbox
  -> Signal Collection Boundary
  -> Provenance Normalization
  -> Evidence Integration Gate + Evidence Memory
  -> Relation/Coverage-Aware Belief Solver
  -> Frame Adequacy Policy
  -> History-Aware Hypothesis Evolution / Frame Expansion
  -> BeliefState t+1
  -> Task-Aware Answer Projection or Next Cycle
```

Run-regime runners continue to own timing, budgets, waiting, and stopping. The
core continues to own evidence admission, belief revision, frame adequacy,
hypothesis evolution, and atomic audit state.

Passive signals do not enter Task Admission or Task Framing as evidence. They
are buffered until a TaskFrame and BeliefState exist. At boundary closure, all
eligible Active and Passive External Signals in the inbox enter the same
provenance and evidence-integration path. Autonomous mode closes the boundary
after its selected probes finish, fail, or time out. Synchronized mode also
honors its configured external-round barrier, so a waiting human or agent turn
cannot be silently skipped.

## 6. Task Admission

### 6.1 Interface

```python
class TaskAdmitter(Protocol):
    def assess(self, input: TaskAdmissionInput) -> TaskAdmissionDecision: ...
```

`TaskAdmissionInput` contains the question, non-evidentiary Task Context,
explicit answer choices or hypotheses when supplied, requested output shape,
available capability descriptors, and sanitized model metadata. It does not
contain passive evidence.

`TaskAdmissionDecision` contains:

```text
attempt_id
status: admitted | needs_reframing | out_of_scope
epistemic_basis
proposed_task_kind | null
answer_contract_outline | null
clarification_questions
reason
model_trace
```

### 6.2 Admission Rule

A task is admitted only when all of the following hold:

- a concrete Answer Contract can be stated;
- at least one uncertainty-bearing, truth-apt, or decision-relevant claim can
  be represented;
- at least one possible probe could distinguish or test those claims;
- an answer can be justified from Belief State plus explicit task criteria.

Broad existential prompts default to `needs_reframing`. Pure generation and
mechanical transformation default to `out_of_scope`.

Explicit MCQ inputs can be admitted deterministically. Unseeded natural
language uses model or recorded adapters with one structured repair. Failure
occurs before TaskFrame or Belief State creation.

### 6.3 Public Result

The public question entry point returns a tagged result:

```text
AdmittedRunResult
NeedsReframingResult
OutOfScopeResult
RunFailedResult
```

`needs_reframing` and `out_of_scope` are successful admission outcomes rather
than provider failures. They create a ledger-visible admission record but no
run, cycle, or belief state.

## 7. TaskFrame and Answer Relationship

TaskFrame remains the immutable task interpretation and gains:

```text
schema_version = v0.2
admission_decision_id
task_kind
answer_relationship: selection | synthesis
answer_contract
initial_hypothesis_frame
framing_method and trace
```

`TaskKind.EXACT_ANSWER` is added for a scalar or short-text answer without an
explicit candidate list.

`AnswerContract` gains a typed answer value constraint:

```text
answer_value_type: choice_label | integer | number | short_text | structured_text
answer_format
required_sections
decision_form
permits_synthesis
```

Relationship rules:

- MCQ: hypotheses are answer candidates; relationship is `selection`.
- Exact answer: hypotheses are provisional answer candidates; relationship is
  `selection`; coverage normally starts open.
- Explanation and diagnosis: hypotheses are explanatory claims; relationship
  can be selection or synthesis according to the contract.
- Design and evidence-informed decision: hypotheses represent assumptions,
  causal claims, risks, and boundary conditions; relationship is synthesis.

The highest-credence claim is therefore not automatically the answer.

## 8. Competition and Coverage

The legacy `HypothesisRelation` is replaced in new writes by two orthogonal
properties:

```text
HypothesisCompetition = exclusive | independent
HypothesisCoverage = exhaustive | open
```

Rules:

- `exclusive + exhaustive`: named active mass sums to one; unresolved mass is
  zero. Explicit MCQ uses this form.
- `exclusive + open`: named active mass plus unresolved alternative mass sums
  to one. Exact-answer candidate discovery uses this form.
- `independent + exhaustive`: each claim carries independent credence and the
  declared claim set is complete for the contract.
- `independent + open`: each claim carries independent credence and the frame
  may add missing claims. Shared unresolved mass is not used because the
  credences are not one categorical distribution.

`unresolved_alternative_mass` is a latent solver reserve, not an `H_other`
hypothesis. It has no answer id and cannot be selected or displayed as an
ordinary hypothesis.

For the first MVP of `exclusive + open`:

```text
initial unresolved mass = 0.50
named initial mass = 0.50 / named hypothesis count
minimum unresolved reserve while coverage remains open = 0.05
```

An initial exclusive-open frame contains one to six validated named
hypotheses. If framing and its single repair cannot produce one valid named
candidate, framing fails before BeliefState creation. Expansion may later grow
the active set to the separate eight-hypothesis runtime limit.

These values are explicit run-policy defaults, recorded in config and ledger,
and do not claim statistical calibration.

## 9. FrameState and Adequacy

TaskFrame preserves initial framing provenance. BeliefState carries the current
versioned `FrameState`:

```text
frame_id
frame_version
parent_frame_version
competition
coverage
active_hypothesis_ids
unresolved_alternative_mass
adequacy_status
revision_reason
trigger_event_ids
revision_count
```

Adequacy states are:

```text
provisional
adequate
challenged
inadequate
expanding
```

Open frames begin provisional. Explicit exhaustive MCQ frames begin adequate.

The deterministic FrameAdequacyPolicy can mark a frame challenged when:

- a signal is poorly explained by all named hypotheses;
- all named candidates receive disconfirming likelihoods;
- a signal points to a value outside the named candidate set;
- a high-quality signal conflicts with the coverage statement;
- unresolved mass exceeds every named candidate.

Any admitted `supports_unresolved` event may trigger exploratory expansion,
because considering a candidate is not equivalent to believing it. A frame is
marked inadequate when either one high-verifiability, strongly unresolved
external/tool event exists or two correlation-distinct, moderately unresolved
events agree. Model reasoning alone can challenge and propose expansion but
cannot by itself establish final adequacy for a contract requiring external
verification.

State transitions are ledger-visible:

```text
provisional -> adequate | challenged
challenged -> adequate | inadequate | expanding
inadequate -> expanding
expanding -> provisional on a new frame version
```

## 10. Evidence Judgment for Open Coverage

Every v0.2 `judge_evidence` request contains:

- raw signal plus complete provenance summary;
- full semantic descriptors for assessed hypotheses;
- competition, coverage, frame version, and relevant rivals;
- probe purpose, target, support, weaken, and reframe conditions;
- prior Evidence Memory novelty and correlation summary;
- allowed evidence, likelihood, and frame-fit enums.

The validated output adds:

```text
likelihoods: {hypothesis_id: likelihood_band}
unresolved_likelihood: likelihood_band | null
frame_fit: explained_by_named | underdetermined | supports_unresolved
unexplained_observation: string | null
evidence_type
interpretation
quality_overrides
```

`unresolved_likelihood` is required for exclusive-open frames and must be null
for every other competition/coverage combination. It updates the latent
unresolved state through the same likelihood-band, quality-weight, provenance,
and correlation-credit policy used for named hypotheses. `frame_fit` remains a
separate adequacy and expansion judgment; it cannot by itself move belief.

Strict validation keeps the two judgments coherent:

- `supports_unresolved` requires a confirming unresolved likelihood;
- `explained_by_named` forbids a confirming unresolved likelihood;
- `underdetermined` requires a neutral unresolved likelihood.

For exclusive-open frames, the solver updates named plus unresolved mass as
one distribution. For independent-open frames, frame fit affects adequacy and
expansion but does not cross-normalize independent credences.

Source policy caps remain deterministic. A model may lower a quality dimension
but cannot increase reliability, independence, or verifiability above the cap
for its epistemic origin.

## 11. Hypothesis Expansion

### 11.1 Interface

```python
class HypothesisExpansionAdapter(Protocol):
    def expand(
        self,
        request: HypothesisExpansionRequest,
    ) -> HypothesisExpansionProposal: ...
```

The request contains TaskFrame, current FrameState, current hypotheses,
Evidence Memory summaries, unexplained signals, answer-value constraints, and
available capabilities.

The proposal may contain only:

```text
candidate statement or typed value
scope
falsifiers
predictions
why the current frame missed it
discovery signal ids
required next probe descriptions
```

The adapter cannot assign ids, priors, posteriors, frame status, or unresolved
mass.

### 11.2 Allocation

Server-assigned ids are stable within the run. Each expansion transfers at
most half of the current unresolved mass, divided uniformly among validated
new candidates, while retaining the open-coverage minimum reserve. Retired
exclusive-open candidates return their remaining mass to unresolved before a
new allocation. Named plus unresolved mass remains one after every transition.

The first policy limits are:

```text
max frame revisions = 3
max active hypotheses = 8
max structured repair attempts per expansion = 1
```

Candidate proposals must be distinct from active and historical statements,
conform to the answer-value type when they are answer candidates, and contain
falsifiers and predictions.

### 11.3 Discovery Evidence

Signals used to create a new hypothesis are recorded as discovery evidence.
They explain why the candidate entered the frame but cannot immediately
increase its posterior. The new hypothesis requires a new probe or a signal
from a different derivation root before it can receive confirming movement.

If expansion fails, the existing Belief State is preserved and the run records
`semantic_evolution_unavailable`. Placeholder hypotheses are forbidden.

## 12. Signal Provenance

Every ExternalSignal gains a structured provenance object:

```text
epistemic_origin
source_identity
provider_model_or_tool_identity
session_id
parent_signal_ids
derivation_root_id
correlation_group
canonical_content_fingerprint
citations
artifact_refs
environment_state_id | null
```

Initial epistemic origins are:

```text
external_observation
retrieved_source
tool_result
model_reasoning
human_input
agent_message
derived_summary
```

A summary derived from a source keeps the source's derivation root. Python or
another deterministic tool may improve verifiability of a derivation but does
not become an independent factual source merely by recomputing the same input.

## 13. Evidence Memory

BeliefState carries a compact, secret-free `EvidenceMemorySnapshot`:

```text
memory_version
accepted_evidence_ids
content_fingerprints
source_content_fingerprints
derivation_roots
correlation_credit
discovery_evidence_ids
counterevidence_ids_by_hypothesis
discard_and_schema_history
```

The ledger remains the full event record. Memory stores only what later cycles
need for deterministic novelty, correlation, lifecycle, and double-use checks.

Cross-cycle rules:

- exact source/root/content repeats are discarded as `duplicate_exact` and
  produce no BeliefUpdate;
- different wording from the same derivation root is marked
  `correlated_restatement`, with zero independence and capped novelty;
- model reasoning from the same provider/model/session shares one correlation
  group;
- correlation credit is bounded per group, hypothesis, and direction, so
  repeated calls cannot create unbounded posterior movement;
- independent accumulation requires distinct source identity and derivation
  root without a parent/derived relationship.

Cycle integration uses prior memory plus a cycle-local working copy, then
commits Evidence Events, Belief Updates, updated memory, and BeliefState
atomically after boundary closure.

## 14. Probe Design

### 14.1 Interfaces

```python
class ProbeDesigner(Protocol):
    def propose(self, context: ProbeDesignContext) -> list[ProbeCandidate]: ...

class ProbeSelector:
    def select(self, candidates, belief_state, budget) -> ProbeSet: ...
```

ProbeDesigner proposes inquiry semantics. ProbeSelector is deterministic and
budget-aware. Model output never assigns posterior or final priority.

`ProbeDesignContext` contains TaskFrame, current FrameState, BeliefState,
Evidence Memory summary, unresolved uncertainty, Answer Contract gaps,
change-my-mind conditions, and Capability descriptors.

### 14.2 Structured Targets

The loose `method="source_tracing"` convention is replaced by typed purposes
and targets:

```text
hypothesis_discrimination
hypothesis_falsification
frame_coverage
source_verification
anomaly_clarification
answer_contract_gap
```

A ProbeDesign declares its purpose, typed target, inquiry goal, expected
observation, support/weaken/reframe conditions, required capability, and
server-assigned cost/priority features.

Initial open-task design must include at least one multi-hypothesis
discriminator or frame-coverage probe. A set of one-hypothesis generic source
tracing probes does not satisfy the contract.

## 15. Capability Registry

Each probe declares one required capability. The initial vocabulary is:

```text
model_reasoning
python_computation
search
document_retrieval
repository_read
test_execution
external_agent_request
human_request
```

`CapabilityDescriptor` records availability, cost class, latency class,
provenance policy, quality caps, and executor adapter identity.

Unavailable capabilities produce an explicit rejected candidate reason. If the
missing capability is material to the Answer Contract, the runner stops with
`external_signal_required` or returns abstention. Model reasoning never
impersonates search, retrieval, test execution, or external verification.

This implementation phase provides registry/config/public interfaces plus
deterministic, recorded, and model-reasoning adapters. It does not add real
search, retrieval, repository mutation, or coding Intervention adapters.

## 16. Candidate Pool Management

Autonomous and Synchronized runners use one shared deep module:

```python
class CandidatePoolManager:
    def build_next_pool(...) -> list[ProbeCandidate]: ...
```

The merge order is fixed:

1. core-produced evolution, anomaly, and source-verification candidates;
2. ProbeDesigner candidates from the latest state;
3. task-aware change-my-mind candidates;
4. remaining unselected candidates.

Candidates are de-duplicated by a semantic fingerprint over purpose, target,
required capability, normalized inquiry goal, and frame version. Temporary ids
are not the identity rule.

`CycleResult.probe_candidates` is required input for the next autonomous and
synchronized cycle. The runners may not omit or silently reorder it.

## 17. History-Aware Hypothesis Evolution

Hypothesis evolution is split into:

1. a deterministic trigger policy that authorizes an operation;
2. a semantic adapter that materializes substantive hypothesis content and
   required follow-up probes.

Supported operations remain spawn, expand, split, reframe, merge, retire, and
reactivate. Trigger decisions use current state plus Evidence Memory, allowing
correlation-distinct counterevidence to accumulate across cycles.

The semantic adapter returns statement/value, scope, falsifiers, predictions,
relation links, frame-failure rationale, discovery evidence ids, and required
next probes. It does not assign beliefs.

Every spawned, expanded, split, or reframed hypothesis must be substantive and
falsifiable and must produce at least one candidate probe. Retired hypotheses
remain in ledger history and can reactivate under new admissible evidence.

One structured repair is allowed. Failure preserves current state and produces
no decorative placeholder.

## 18. Task-Aware Projection

### 18.1 Interface

```python
class ProjectionGenerator(Protocol):
    def project(self, input: ProjectionInput) -> AnswerProjection: ...
```

ProjectionInput contains Task Admission, Answer Contract, answer relationship,
current FrameState, BeliefState, accepted evidence summaries and origins,
Evidence Memory, missing capabilities, unresolved uncertainty, and stop reason.

AnswerProjection v0.2 contains:

```text
projection_mode: selection | synthesis | abstention
answer
typed_answer_value | null
current_best_hypothesis | null
basis_hypotheses
completed_contract_sections
unmet_contract_sections
main_evidence_events
source_citations
material_uncertainties
missing_capabilities
change_my_mind_condition
```

### 18.2 Selection

MCQ remains a deterministic selection projection over an explicit exhaustive
choice set. Benchmark contracts may require a top label even when confidence is
low, with uncertainty preserved in the output.

Open exact-answer selection requires all of:

```text
frame adequacy = adequate
top posterior >= 0.60
top versus runner-up margin >= 0.15
unresolved mass <= 0.20
typed value conforms to Answer Contract
no unexplained high-quality anomaly
```

The thresholds are configurable, ledger-visible MVP policies rather than
calibrated statistical claims.

### 18.3 Synthesis

Explanation, design, and evidence-informed decision tasks synthesize from one
or more belief-backed claims. The generated result must satisfy every required
Answer Contract section, distinguish model reasoning from external evidence,
state missing capabilities, and cite only sources present in Signal Provenance.

### 18.4 Abstention and Failure

Abstention is required when frame adequacy is insufficient, unresolved mass is
too high, material capabilities are unavailable, signals conflict, or the
Answer Contract cannot be met.

`project_open_answer` permits one structured repair. A second failure preserves
BeliefState and returns `answer_unavailable`; it never falls back to a generic
top-hypothesis sentence.

Projection cannot create Evidence Events, modify posterior values, or invent
citations.

## 19. Runner and Progress Semantics

The autonomous lifecycle adds:

```text
task_admission_started
task_admission_completed
task_framing_started
task_framing_completed
initialization_completed
probe_design_started
probe_design_completed
probe_set_planned
capability_unavailable
external_signal_required
hypothesis_expansion_started
hypothesis_expansion_completed
answer_projection_started
answer_projection_completed
answer_projection_failed
run_completed | run_failed
```

No progress event may expose a BeliefState before initialization. Admission,
framing, expansion, evidence, and projection failures use the shared secret-safe
diagnostic path.

Both autonomous and synchronized runners preserve pre-cycle and in-cycle
Passive External Signals until their Signal Collection Boundary closes. A
passive-only synchronized cycle remains valid: it has an empty selected Probe
Set, processes the buffered signals through the same gate, and emits a normal
Belief State Projection.

Stop semantics distinguish:

```text
answer_contract_satisfied
max_cycles
no_valuable_probes
external_signal_required
frame_revision_budget_exhausted
answer_unavailable
```

An unresolved open task stopped by budget projects abstention. An explicit MCQ
continues to satisfy its required-label contract when the run reaches a normal
terminal state.

## 20. WebUI, SDK, and Configuration

The WebUI exposes and renders:

- Task Admission status and clarification questions;
- Answer Contract and answer relationship;
- frame competition, coverage, adequacy, and unresolved mass;
- probe purpose and required capability;
- signal epistemic origin and correlation status;
- evolution and frame-version history;
- projection mode, completed/unmet sections, and missing capabilities.

Credentials remain request-scoped and page-memory-only. The current API remains
available for compatible MCQ/open requests, while responses declare schema
version and tagged result type.

The package root exports TaskAdmitter, admission result types, FrameState,
EvidenceMemorySnapshot, ProbeDesigner, ProbeSelector, CandidatePoolManager,
CapabilityRegistry, semantic evolution adapters, ProjectionGenerator, policy
configs, and migration helpers. External projects do not need private imports.

Run config gains explicit admission, open-coverage, frame-adequacy,
correlation-credit, capability, expansion, and projection policies. Raw keys
remain forbidden in JSON config.

## 21. Failure Semantics

Failure is local and state-preserving whenever possible:

- unavailable capability -> visible rejection, external signal request, or
  abstention;
- expansion failure -> current BeliefState plus
  `semantic_evolution_unavailable`;
- projection failure -> current BeliefState plus `answer_unavailable`;
- revision budget exhaustion -> abstention;
- invalid provider output -> one repair then fail closed;
- exact duplicate -> discarded event and no BeliefUpdate;
- correlated saturation -> no remaining evidence credit.

No path silently creates generic hypotheses, marks an open frame exhaustive,
turns proposals into evidence, or presents unverified model reasoning as an
external observation.

## 22. Security and Reproducibility

All task admission, framing, repair, expansion, probe design, evidence, and
projection payloads use the shared secret detector and redactor. Provider keys
remain only in request-scoped provider construction.

Evidence Memory contains fingerprints and refs, not raw secret-bearing text.
Provenance and correlation metadata must be recursively secret-free before
ledger or shareable artifact output.

Recorded fixtures match structured task identity, never raw restricted question
text. HLE or other gated benchmark questions are not committed as regression
fixtures. Self-authored semantic equivalents cover the same architecture paths.

Every policy default, prompt/schema version, capability snapshot, and frame
revision is ledger-visible and included in experiment artifacts where present.

## 23. Migration and Compatibility

New writes use schema v0.2. Legacy serialized states are accepted only through
an explicit migration function:

```text
HypothesisRelation.EXCLUSIVE_EXHAUSTIVE
  -> competition = exclusive
  -> coverage = exhaustive
  -> unresolved mass = 0
```

Legacy independent states map to independent competition and their documented
coverage limitation as `coverage = open`; v0.1 had no representation for a
declared exhaustive independent claim set. Migration never infers
open/exhaustive status from ids, count, or posterior values.

Explicit MCQ choice ids, uniform priors, answer labels, and deterministic
projection remain compatible. Recorded v0.1 frames are migrated and validated
strictly; malformed native structures continue to fail closed.

The old `HypothesisRelation` symbol remains readable during the v0.2 migration
window but is deprecated for new construction. New code uses competition and
coverage explicitly.

## 24. Testing Strategy

### 24.1 Contract Tests

Tests must prove:

- admission can return admitted, needs_reframing, and out_of_scope without
  premature BeliefState creation;
- exact-answer frames use exclusive-open semantics;
- an invalid zero-candidate exclusive-open frame fails before BeliefState
  creation;
- unresolved likelihood is present only for exclusive-open frames and agrees
  with frame fit;
- named plus unresolved mass remains one across update, retirement, and
  expansion;
- all named candidates can be rejected without forced selection;
- expansion can add a missing candidate under bounded reserve allocation;
- discovery evidence cannot confirm the candidate it generated;
- exact duplicates and same-root restatements are neutral across cycles;
- correlation groups enforce bounded cumulative evidence credit;
- evidence judgment receives complete semantics, provenance, frame fit, and
  memory summary;
- unavailable capability is visible and never impersonated;
- pre-cycle and in-cycle passive signals wait for boundary closure and never
  enter admission or framing as evidence;
- initial open probes discriminate rivals or test frame coverage;
- core-produced candidates enter the next autonomous and synchronized cycles;
- semantic evolution uses history and never generates placeholders;
- projection modes obey task and adequacy rules;
- synthesis satisfies every required Answer Contract section;
- projection cannot mutate belief or invent citations;
- v0.1 migration preserves explicit MCQ behavior.

### 24.2 Recorded Vertical Slices

Self-authored recorded fixtures cover:

- MCQ regression;
- open exact answer with a missing initial candidate;
- independent claim verification;
- design synthesis;
- an existential prompt requiring reframing;
- an out-of-scope generation task;
- an unavailable capability;
- active-only, passive-only, and mixed cycles.

Each slice runs through the public runner and asserts ledger-visible admission,
framing, probes, provenance, evidence, memory, evolution, and projection.

### 24.3 Verification Gates

Completion requires:

- full Python suite;
- WebUI stream suite;
- explicitly enabled real Docker isolation suite;
- recorded public-runner end-to-end slices;
- five self-authored live-provider smoke questions;
- recursive secret and artifact leak scans;
- two independent whole-branch reviews with no unresolved Critical or
  Important findings.

## 25. Delivery Milestones

The implementation is delivered in independently reviewed milestones:

1. integrate the approved correction baseline into `main`, push the frozen
   baseline, and branch for kernel completion;
2. implement Task Admission, schema v0.2, exact-answer, competition, coverage,
   FrameState, and migration;
3. implement Signal Provenance, Evidence Memory, frame-fit judgment, and the
   coverage-aware solver;
4. implement ProbeDesigner, typed targets, Capability Registry,
   ProbeSelector, and CandidatePoolManager;
5. implement history-aware semantic evolution, frame expansion, and
   ProjectionGenerator;
6. integrate autonomous/synchronized runners, progress events, WebUI, SDK,

7. complete recorded vertical slices, live smoke, full verification, reviews,

Every milestone ends with focused tests, a public-runner vertical slice where
applicable, full regression, diff checks, and independent review. Unit tests
alone cannot close a milestone.

## 26. Future Validation Roadmap

No new benchmark adapter is implemented in this phase. The existing HLE
text-MCQ harness remains available but is not formally executed here. The
architecture roadmap records:

1. SWE-bench Verified as the first primary coding-agent validation after a
   separate Intervention layer exists;
2. Terminal-Bench after general terminal capabilities and environment-change

3. RE-Bench as a later high-cost autonomous AI R&D validation;

Comparative runs must hold model, tools, environment, token/time/cost budget,
and retry policy constant across the BayesProbe and reference arms.

## 27. Explicit Non-Goals for This Phase

- coding repository mutation or Intervention execution;
- real search, document retrieval, or browser adapters;
- SWE-bench, Terminal-Bench, or RE-Bench adapters;
- a private BayesProbe benchmark;
- formal HLE execution;
- a philosophical perspective-state engine;
- SQLite, distributed orchestration, provider registry, or production
  telemetry backend;
- claims of calibrated Bayesian probabilities or methodological superiority.

## 28. Definition of Done

The epistemic kernel completion is done only when:

1. inadmissible tasks create no BeliefState and return useful reframing or
   scope results;
2. answer candidates and belief claims are no longer conflated;
3. exact-answer tasks can preserve and expand unresolved alternative space;
4. a missing initial candidate can be discovered without self-confirmation;
5. repeated or correlated reasoning cannot accumulate as independent evidence;
6. Evidence Judgment sees full semantics, provenance, and memory;
7. first-cycle open probes are discriminative or coverage-seeking;
8. all core-produced candidates reach the next cycle;
9. semantic evolution is substantive and history-aware;
10. selection, synthesis, and abstention satisfy their task contracts;
11. MCQ, active/passive/mixed, synchronized, provider, Docker, security, and
    artifact behavior remain green;
12. live smoke and independent review confirm the public WebUI and SDK paths;
13. architecture documentation distinguishes implemented capability from
    future coding Intervention and benchmark validation.
